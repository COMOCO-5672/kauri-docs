# Visual Documentation Standard

这个仓库的项目文档必须图文并茂。文字负责解释判断，图负责让读者先建立结构感。

## 每个项目必须包含的图

1. 整体架构图：展示顶层模块、边界和依赖方向。
2. 主流程图：展示一个典型任务从输入到输出经过哪些模块。
3. 关键子系统图：展示最复杂或最容易误解的内部机制。
4. 数据结构/状态流图：当项目依赖元数据、状态机、缓存、任务队列时必须补充。
5. 缺陷/风险图：对不完善场景给出范围图或决策树，避免只列文字。

## 图的形式

- 首选 Mermaid：流程图、时序图、状态图、模块关系图。
- 对协议、文件格式、内存布局可使用 ASCII 图或 SVG。
- 对 UI、渲染、媒体画面、硬件拓扑可使用 PNG/SVG。
- 一张图只讲一个问题，不把所有信息塞进一张巨图。

## 文档写法

- 每张图前用 1-2 句话说明它回答什么问题。
- 每张图后列出对应源码入口，精确到文件和函数名。
- 图中节点名称要和源码模块/函数/数据结构能对应起来。
- 复杂表格前必须先有图，否则读者很难理解表格维度。
- 高级经验文档必须配合 `> [!IMPORTANT]`、`> [!WARNING]`、`> [!TIP]`、`> [!NOTE]` 标记，让核心合同、风险和工程建议从普通正文中区分出来。
- 图和表要服务于工程决策：读者应能看出正常路径、异常路径、fallback 位置和需要打印的关键字段。
